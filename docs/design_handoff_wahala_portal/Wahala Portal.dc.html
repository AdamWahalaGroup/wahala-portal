13 — Build note Next.js App Router · server components vs client islands

Component spec — mapping to the stack

Pages are React Server Components on Cloudflare Workers: they fetch from D1, compute the role-and-state action set on the server, and stream HTML. Only the small interactive pieces ship JS as client islands — keeping the Workers bundle lean and the security model server-authoritative.

Routes → page type
/loginRSC shell · island: email form (idle/sending/sent/error)
/dashboardRSC · role-branched (staff vs client). Static lists, no island.
/dashboard/projects/:idRSC · stage list. island: "New stage" drawer (staff)
/dashboard/stages/:idRSC renders state + history. islands: the action set, confirm dialogs, acceptance checklist, pay button
API the islands call
POST /api/stages/:id/:action
send_quote · approve_quote · reject_quote
mark_paid · start_work · deliver
accept · request_revision
Components → server or island
StatusBadge, Stepper, Money, HistoryTimelineserver
WaitingOn, VisibilityMarker, PeopleCardserver
ActionBar (the allowed-action set)island
ConfirmDialog (pay / accept / reject / co-sign)island
LineItemEditor (quote builder)island
AcceptanceChecklist, FileUpload, MessageComposerisland
Server-authoritative rules
The RSC computes the action set from role + state + threshold and tenant-scopes every query. Islands render only what they're handed — they never decide permissions or filter visibility client-side. The pay-gate, co-sign, and internal/visible split are enforced before HTML is sent.

Two objects, two records of work, one loop.

Sales and Projects are not two apps — they are two lenses over the same Account. Nothing is ever "converted": a person stays one Contact record forever, an org stays one Account record forever. Only their state changes.
